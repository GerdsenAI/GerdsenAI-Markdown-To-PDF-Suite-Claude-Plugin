# Software Architecture Blueprint Reference

Structure template, research facets, technology evaluation criteria, diagram requirements, and quality standards for software architecture blueprint reports.

This reference activates when the report type is **Software Architecture Blueprint**. It supplements the base research-report-reference — all citation, quality, and multi-document standards from that reference still apply.

### Section Guidelines

- **Executive Summary**: Summarizes the recommended stack, key architectural decisions, and estimated timeline. No citations — it synthesizes.
- **Requirements**: Functional requirements use MoSCoW prioritization. Non-functional requirements must have specific measurable targets (not "fast" but "p95 < 200ms").
- **Technology Stack**: Every recommendation includes a comparison table with at least 2-3 alternatives evaluated. Rationale addresses why over alternatives, not just why the choice is good.
- **Database Schema**: ER diagrams split by domain boundary if more than 10 entities. Every entity in the diagram must have a corresponding definition table.
- **API Design**: Endpoint tables include auth requirements. Key flows shown as sequence diagrams. Naming conventions must be internally consistent throughout.
- **Authentication**: Auth flow diagrams cover the full lifecycle including token refresh and session expiry.
- **Infrastructure**: Cost-conscious — shows monthly costs at the stated scale, not just "it scales."
- **Implementation Roadmap**: Gantt chart phases correspond to document sections. Milestones are concrete deliverables, not vague goals.
- **Cost Estimation**: Covers compute, database, third-party services, and development labor. Monthly figures at stated scale.
- **Risk Assessment**: Every risk has a mitigation strategy. Vendor lock-in analysis for each major technology choice.

## Research Facets

When in Blueprint mode, launch 5-6 sub-agents in parallel, each covering one facet:

| Facet | What to Research | Key Data Points to Return |
|-------|-----------------|--------------------------|
| **Framework & Language Ecosystem** | Latest stable versions, GitHub activity, downloads, LTS schedules, migration paths, breaking changes, TypeScript/type safety support | Name, version, release date, stars, weekly downloads, license, last major breaking change, LTS end date |
| **Database & Data Layer** | Engine comparisons for use case (relational/document/graph/time-series), hosted vs self-managed, pricing at scale, ORM/driver quality, migration tooling | Comparison table with performance benchmarks, pricing tiers, ecosystem maturity, hosted options |
| **Authentication & Security** | Auth provider comparisons (Auth0, Clerk, Supabase Auth, Firebase Auth, Keycloak, WorkOS), compliance requirements, encryption standards, SSO support | Pricing by MAU, feature comparison table, SSO support, compliance certifications, free tier limits |
| **Infrastructure & DevOps** | Hosting comparisons (AWS/GCP/Azure/Vercel/Fly.io/Railway/Cloudflare), CI/CD tooling, container orchestration, cost modeling, monitoring stack | Monthly cost at stated scale, cold start times, scaling limits, free tier details, region availability |
| **API & Integration Patterns** | REST vs GraphQL vs gRPC suitability, real-time options (WebSocket/SSE/MQTT), API gateway options, rate limiting strategies, SDK generation | Latency benchmarks, tooling ecosystem, complexity tradeoffs, client library availability |
| **Community & Documentation Quality** | Stack Overflow tag activity, Discord/Slack community sizes, docs completeness, tutorial ecosystem, corporate backing, job market demand | Member counts, question response times, docs last-updated dates, backing company stability, job postings |

Each sub-agent prompt must include:
- The specific facet and the user's project context (app type, scale, constraints)
- Instructions to return: product name, latest version, release date, GitHub stars, weekly downloads, license, limitations, comparison with 2-3 alternatives
- Instructions to track ALL source URLs with title, author, date, and access date

## Technology Evaluation Table Format

Every technology recommendation must include a standardized comparison table:

| Criterion | How to Source |
|-----------|--------------|
| Latest stable version (semver) | GitHub releases, official docs |
| Release date of latest version | GitHub releases |
| GitHub stars | GitHub repository |
| npm/PyPI/crates.io weekly downloads | Package registry stats |
| Open issues count | GitHub Issues |
| Documentation quality (Excellent/Good/Fair/Poor) | Manual assessment of docs site |
| Community size (Discord/Slack members) | Community platforms |
| Corporate backing | Official announcements, about pages |

## API Specification Format

### Endpoint Table Format

```markdown
| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | /api/v1/auth/login | No | Authenticate user, return JWT |
| GET | /api/v1/users/:id | Bearer | Get user profile |
| PUT | /api/v1/users/:id | Bearer + Owner | Update user profile |
| DELETE | /api/v1/users/:id | Bearer + Admin | Delete user account |
```

### Request/Response Schema Format

````markdown
**POST /api/v1/auth/login**

Request:
```json
{
  "email": "string (required)",
  "password": "string (required, min 8 chars)"
}
```

Response (200):
```json
{
  "token": "string (JWT)",
  "refresh_token": "string",
  "expires_in": 3600,
  "user": {
    "id": "uuid",
    "email": "string",
    "role": "admin | user | viewer"
  }
}
```

Error (401):
```json
{
  "error": "INVALID_CREDENTIALS",
  "message": "Email or password is incorrect"
}
```
````

### Error Code Table Format

```markdown
| Code | Name | HTTP Status | Description |
|------|------|-------------|-------------|
| INVALID_CREDENTIALS | Invalid Credentials | 401 | Email or password incorrect |
| TOKEN_EXPIRED | Token Expired | 401 | JWT has expired, use refresh token |
| FORBIDDEN | Forbidden | 403 | Insufficient permissions for this action |
| NOT_FOUND | Not Found | 404 | Requested resource does not exist |
| RATE_LIMITED | Rate Limited | 429 | Too many requests, retry after header value |
```

## Blueprint Quality Checklist

All standard research-report-reference quality checks apply, plus these architecture-specific checks:

### Technology Recommendations
- [ ] Every technology recommendation includes: latest version, release date, and at least one quantitative metric (stars, downloads, or benchmark)
- [ ] Every recommendation addresses: why over alternatives, community health, documentation quality, long-term viability
- [ ] Comparison tables include at least 2-3 alternatives for each major technology choice
- [ ] No technology recommendation without stated rationale

### Database & API Consistency
- [ ] ER diagram entities match Database Schema section text (no phantom entities, no missing entities)
- [ ] API endpoints include: method, path, auth requirement, request schema, response schema, error codes
- [ ] API naming conventions are internally consistent (all kebab-case or all camelCase, not mixed)
- [ ] Request/response schemas use consistent type annotations

### Architecture Coherence
- [ ] C4/component diagrams reference the same systems named in Technology Stack
- [ ] Gantt chart phases correspond to document sections and milestones
- [ ] Infrastructure diagrams match the services listed in Cost Estimation
- [ ] Auth flow diagrams cover login, token refresh, and session expiry

### Requirements & Planning
- [ ] Non-functional requirements have specific measurable targets (not "fast" but "p95 < 200ms")
- [ ] Functional requirements use MoSCoW or similar prioritization
- [ ] Cost estimation provides monthly figures covering: compute, database, third-party services, and development labor
- [ ] Implementation roadmap has concrete deliverables per phase, not vague goals

### Security
- [ ] Security section addresses applicable OWASP Top 10 items
- [ ] Compliance requirements from intake are addressed in Security Considerations
- [ ] Auth provider recommendation includes security certifications

### Testing
- [ ] Testing strategy specifies tool names and versions
- [ ] Test pyramid defines target coverage percentages

## Multi-Document Split Points

If a blueprint exceeds approximately 50 pages, split into two parts:

- **Part 1: Requirements & Architecture** — Sections: Executive Summary, Project Overview & Requirements, Technology Stack, System Architecture, Database Schema, API Design
- **Part 2: Operations & Planning** — Sections: Authentication & Authorization, Infrastructure & Deployment, Security, Testing, Implementation Roadmap, Cost Estimation, Risk Assessment

Each part gets its own Executive Summary scoped to its content. Citation numbering is continuous across parts. Cross-references between parts use the format: "See Part 2: Infrastructure & Deployment, Section 2.1."
